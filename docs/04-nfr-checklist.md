# Non-Functional Requirements (NFR) Checklist

## Базові вимоги
- [ ] **UTC + ISO 8601:** Усі дати та час (`arrived_at`, `storage`, `created_at`) у контрактах API, базі даних та логах фіксуються виключно у форматі ISO 8601 у часовому поясі UTC.
- [ ] **Стабільні коди помилок:** Будь-яка бізнес-помилка (наприклад, `RuleResult.Failed`) повертається на клієнт не просто текстом, а машинозчитуваним кодом (`slot_unavailable`, `location_mismatch`, `wrong_parcel_status`).
- [ ] **Ідемпотентність запитів:** Завдяки інтеграції MassTransit та Outbox Messages (у поєднанні з `ProcessedLog`), повторні спроби `AcceptParcel` не призводять до дублювання чи зависання системи.
- [ ] **Event-Driven Architecture:** Усі зміни станів (наприклад, прийняття посилки) не блокують головний потік виконання; вони реєструються в Outbox і обробляються у фоновому режимі (Message Queue).

## Акцент: Performance-first (Twist B)
- [ ] **Легкі DTO / CQRS:** Читання даних (Queries) відокремлено від запису (Commands). Відповіді містять лише необхідні поля без вивантаження усього графа сутностей EF Core.
- [ ] **Pagination (Пагінація):** Усі методи отримання списків (наприклад, `GetAllAsync` або пошук вільних слотів) повинні підтримувати `offset/limit` для уникнення перевантаження пам'яті.
- [ ] **Batching:** Запис логів та івентів у базу даних здійснюється пакетами (`MaxBatchSize(1000)` у Npgsql) для мінімізації кількості підключень до бази даних.
- [ ] **Відсутність зайвих JOIN-ів:** Статус посилки зберігається безпосередньо як Value Object / Окрема колекція (StatusHistory) з оптимізованими індексами для швидкого пошуку (`CurrentStatus`).
