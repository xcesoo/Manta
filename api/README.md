# Документація до практичної роботи: Manta Logistics API

Даний проєкт є серверною частиною системи логістики, реалізованою на платформі .NET 9. Архітектура системи базується на принципах Event-Driven Design, DDD, паттернах CQRS та Outbox для забезпечення надійної асинхронної обробки даних.

## Технологічний стек

* **Runtime:** .NET 9 (ASP.NET Core)
* **Database:** PostgreSQL (Entity Framework Core)
* **Message Broker:** RabbitMQ (MassTransit)
* **Communication:** SignalR (Real-time сповіщення)
* **Containerization:** Docker, Docker Compose

## Інструкція із запуску системи

Система повністю контейнеризована, що дозволяє розгорнути інфраструктуру та прикладний код однією командою без необхідності встановлення локальних залежностей.

1. Переконайтеся, що в системі встановлено та запущено **Docker Desktop** (або Docker Engine з Docker Compose).
2. Перейдіть у кореневу директорію проєкту, де знаходиться файл `compose.yaml`.
3. Виконайте команду для збірки та запуску контейнерів:
   ```bash
   docker-compose up --build -d
   ```
4. Після завершення запуску система автоматично:
    * Підніме PostgreSQL та RabbitMQ.
    * Проведе створення нової структури таблиць.
    * Виконає ініціалізацію тестових даних, включаючи системного користувача.

## Тестування за допомогою Postman

Для верифікації функціональності бізнес-логіки (зокрема валідації місткості відділень — Twist A) підготовлено колекцію запитів.

**Шлях до файлу колекції:** `/mock/requests.postman_collection.json`

### Порядок виконання тестів:
1. Імпортуйте файл колекції у Postman.
2. Відкрийте **Collection Runner** для папки `Manta API`.
3. Запустіть виконання колекції.

### Очікувані результати:
* Усі запити на створення та авторизацію мають повернути статус **200 OK** або **202 Accepted**.
* Запит `Accept Parcel_4 Twist A` має завершитися помилкою **409 Conflict**. Це підтверджує роботу механізму синхронної валідації (Fast-Fail), який блокує додавання посилки до відділення, ліміт місткості
